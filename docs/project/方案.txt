import win32com.client as win32
import os

def add_word_comment(doc_path, comment_text, author="Reviewer"):
    """
    为 Word 文档添加批注
    
    参数:
    doc_path (str): Word 文档路径
    comment_text (str): 批注内容
    author (str): 作者名称（默认为 "Reviewer"）
    """
    # 启动 Word 应用
    word = win32.Dispatch("Word.Application")
    word.Visible = True  # 设置为 True 以便查看操作过程
    
    # 打开文档
    doc = word.Documents.Open(os.path.abspath(doc_path))
    
    try:
        # 创建批注（添加到文档开头）
        comment_range = doc.Range(0, 0)  # 0,0 表示文档开头
        
        # 添加批注（不包含Author参数）
        comment = doc.Comments.Add(
            Range=comment_range,
            Text=comment_text
        )
        
        # 设置批注作者
        comment.Author = author
        
        # 保存文档
        new_doc_path = os.path.splitext(doc_path)[0] + "_with_comment.docx"
        doc.SaveAs(new_doc_path)
        
        print(f"批注已添加，文档已保存至: {new_doc_path}")
        return new_doc_path
    
    finally:
        # 关闭文档和 Word
        doc.Close()
        word.Quit()

一、目标与边界
- 目标：基于用户输入意图与“专家视角”，自动对Word长文档进行修订并输出带修订痕迹的Word版本。
- 输出：修订版docx（开启修订痕迹），以及可选的修订摘要/变更说明（便于人工复核）。
- 边界：不提供代码实现；方案以可落地的技术与流程设计为主。

二、总体架构（四层）
1) 交互层：采集用户意图、专家角色、修订范围、约束规则（如不改数据、保留格式）。
2) 文档层：读取Word结构（标题层级、样式、表格、图表、附录），建立“文档地图”。
3) 智能体层：LangChain 1.0 Deep Agents（agent harness + 中间件），主智能体负责任务规划、进度控制与质量汇总；子智能体按职责分工。
4) 执行层：通过Python修订引擎启用修订/批注，完成对段落/表格/标题的改写、增删与批注，并保存中间版本。

三、技术实现映射（Deep Agents为主）
1) Agent harness与默认中间件
- 规划与进度：内置TodoListMiddleware提供write_todos，适配多步修订计划与进度跟踪。
- 文件系统：FilesystemMiddleware提供ls/read_file/write_file/edit_file等能力，用于长文档的分段读取与结果落盘。
- 子智能体：SubAgentMiddleware通过task隔离复杂子任务，主智能体只接收结果，降低上下文膨胀。
2) 工具体系与可调用能力
- Word修订、文档解析、结构化摘要等由自定义工具承载，统一纳入Deep Agents工具调用循环。
- 对长文档段落读写使用offset/limit式读取与分段保存策略，降低一次性上下文占用。
3) 文档修订引擎（Python）
- 采用win32com作为主修订引擎，支持批注/修订标记等能力，适配docx修订输出。
- python-docx作为轻量级兜底方案，仅用于“新增内容/模拟修订”的场景。
- 统一通过工具接口封装修订引擎能力，确保与Deep Agents的工具调用循环对齐。
4) 文件系统与存储后端
- StateBackend用于线程内临时文件（章节摘要、修订理由）。
- FilesystemBackend用于真实落盘（原文与修订版docx），必须限定root_dir并启用virtual_mode。
- StoreBackend用于跨会话记忆（专家规则、偏好），通过CompositeBackend将/memories/路由到持久化存储。
5) 记忆与上下文控制
- memory参数加载AGENTS.md，作为可复用的专家规则与审阅标准。
- 大结果自动淘汰到文件系统（tool_token_limit_before_evict），主智能体按需回读。
- 长会话自动摘要与保留近期上下文，保证多轮修订稳定性。
6) 子智能体编排
- 子智能体按“结构解析/专家修订/一致性核查/语言润色”拆分，隔离上下文与工具调用。
- 子智能体为无状态执行，只回传最终报告，便于主智能体统一裁决。
7) 人机协同与审批
- interrupt_on对高风险工具启用审批点（覆盖保存、批量替换、删除等）。
- 审批支持批准/编辑/拒绝；需检查点器保证中断后的恢复与追溯。
8) 模型与提示控制
- model支持provider:model形式，确保具备工具调用能力。
- system_prompt注入专家视角、禁改规则与改写边界；需要时启用提示缓存降低成本。
9) 技能与模板化
- 将行业规范、审阅清单、专家模板封装为SKILL.md，按需加载（渐进式披露）。
- 技能来源支持文件系统加载或运行时注入，后加载版本覆盖前加载版本。
10) 安全与合规边界
- FilesystemBackend在生产环境需谨慎使用，优先沙箱/受限路径。
- 对敏感文件与密钥设置隔离，不进入可访问根目录。
11) 可选增强（参考langchian1精简版.md）
- 需要更强可靠性与合规时，可引入重试、回退、限流、PII防护等中间件能力。
- 结构化输出用于修订摘要与变更清单的统一格式化输出。

四、关键能力设计
1) Word修订模式控制
- 使用Python修订引擎启用修订；任何改动均以修订痕迹或批注形式输出。
- win32com为主要修订实现，python-docx作为轻量兜底（不支持原生修订，仅可模拟新增）。
- 对表格、标题、页眉页脚分别处理，确保样式不被破坏。
2) 长文档处理
- 按结构分段（章节/小节/表格区块），逐段修订、逐段保存。
- 对每段生成“局部摘要 + 修改理由”，写入临时文件用于上下文压缩与回溯。
3) 专家视角建模
- 将“专家角色描述 + 审阅标准 + 禁改规则”写入系统提示和任务清单。
- 支持多维约束：风格（正式/学术）、语气（中立/谨慎）、领域术语一致性。
4) 证据与一致性
- 对事实性内容设置“保守改写”策略；数值与结论需保留或标记来源不明的风险。
- 全文一致性回扫：术语统一、缩略语定义一致、图表与正文一致。
5) 变更可追踪
- 生成修订摘要：每章变更类型（措辞/结构/逻辑/合规），便于人工快速审查。
6) 人机协同审批
- 对敏感操作启用中断审批（如覆盖保存、批量替换、删除内容）。
- 支持“批准/编辑/拒绝”三类决策，形成可审计的审批链路。
7) 上下文压缩与回读
- 大结果自动淘汰到文件系统，按需回读，避免上下文窗口溢出。
- 对话历史自动摘要，保证长会话下的稳定性与可持续性。
8) 技能化知识与模板
- 将专家规则、修订规范、行业模板沉淀为SKILL.md，按需加载。
- 通过渐进式披露减少系统提示体积，提高长文档任务的可控性。
9) 记忆结构与长期知识
- 通过AGENTS.md/持久化路径沉淀用户偏好、禁改清单与风格要求。
- 定义记忆结构与清理机制，避免长期记忆无限膨胀。

五、智能体方案（LangChain 1.0 Deep Agents）
1) Agent harness与中间件
- 内置规划中间件：使用write_todos维持多步计划、优先级与进度更新。
- 内置文件系统中间件：提供ls/read_file/write_file/edit_file，用于长文档上下文卸载与回溯。
- 内置子智能体中间件：通过task委派复杂任务，隔离上下文，减少主智能体膨胀。
- 工具大结果淘汰：超阈值结果写入文件并引用，主智能体按需回读。
2) 主智能体职责
- 任务规划：生成修订计划、优先级与风险点清单，并随进展动态更新。
- 全局一致性：跨章节冲突检测、术语统一、风格统一。
3) 子智能体建议分工
- 结构分析子智能体：读取目录与样式，输出“文档地图”。
- 专家修订子智能体：按专家视角执行内容修订。
- 事实/一致性子智能体：检查前后矛盾、数值变动、图表引用问题。
- 语言润色子智能体：在不改变事实的前提下提升可读性。
- 子智能体可配置不同模型/工具/中间件，但只回传最终结果（无多轮中间上下文）。
4) 文件系统与存储后端
- StateBackend：线程内临时存储，适合章节级中间结果。
- FilesystemBackend：真实磁盘读写，适合Word原文与修订版落盘。
- StoreBackend：跨会话持久化记忆，适合长期的专家规则与偏好。
- CompositeBackend：将/memories/路由至StoreBackend，其余路径保持临时态。
- FilesystemBackend需限定root_dir并启用virtual_mode，生产环境建议使用SandboxBackend。
5) 记忆与上下文控制
- 使用AGENTS.md作为记忆文件沉淀专家视角、禁改规则与审阅标准。
- 大结果自动淘汰：工具结果超阈值时自动写入文件并引用，避免上下文溢出。
- 章节摘要/修订理由写入临时文件，主智能体按需read_file回读。
- 建议在系统提示中声明记忆结构（如/memories/preferences.txt等）并定期清理旧数据。
- 对话历史自动摘要，触发阈值后保留近期上下文，保障超长会话稳定。
6) 模型与系统提示
- 默认模型可替换为provider:model，保障与工具调用能力匹配。
- 系统提示需显式注入专家规则、禁改清单与改写边界。
- 对超长系统提示可启用提示缓存以降低成本与延迟（特定模型支持）。
7) 人机协同与中断
- 通过interrupt_on为关键工具设置审批点，避免高风险误操作。
- 启用审批需持久化检查点，以保证中断后的恢复与追溯。
- 子智能体可覆盖主智能体的审批配置，细化不同环节的管控。
8) 技能与模板化
- 将行业规范、审阅流程、格式准则封装为技能目录，按需加载。
- 多来源技能按顺序覆盖（后加载优先），便于版本迭代。
9) 观测与追踪
- 通过代理名称元数据区分主/子智能体结果，便于问题定位与审计。

六、流程（端到端）
1) 输入阶段：用户提供目标、专家视角、修订范围与禁改清单。
2) 解析阶段：生成文档地图（章节、样式、表格、图表、附录）。
3) 规划阶段：write_todos生成修订计划与风险提示。
4) 修订阶段：逐章节调用子智能体执行修订，输出带修订痕迹的改动。
5) 审批阶段（可选）：对关键改动触发人工批准/编辑/拒绝。
6) 一致性回扫：统一术语、修正前后不一致、优化衔接。
7) 输出阶段：交付修订版Word + 修订摘要（可选）。

七、性能与稳定性
- 分段处理 + 中间保存，避免长文档在一次处理内耗尽上下文。
- 启用工具结果淘汰阈值，降低大段落结果对上下文窗口的压力。
- 自动摘要对话历史，避免超长交互导致上下文崩溃。
- 对Word自动化设置超时与重试，降低卡死风险。
- 支持断点续修：某章失败不影响其他章节。
- 中断/取消的工具调用可被修复，防止消息链断裂导致误判。

八、质量与验收
1) 质量检查清单
- 修订痕迹存在且可接受/可拒绝。
- 数值与结论未被无依据修改。
- 术语与风格一致，语义不偏离原意。
- 若启用审批，需保留审批记录与最终决策痕迹。
2) 验收标准
- 修订版docx可在Word中完整显示修订痕迹。
- 修订摘要能定位到章节与修改类型。
- 用户意图在全文中被体现（如强调风险、提升专业度等）。

九、风险与对策
- Word自动化不稳定：使用分段保存与重试策略。
- 模型“过度改写”：设置严格禁改规则与保守修订策略。
- 图表/表格格式破坏：单独处理，必要时仅批注不改写。
- 文件系统风险：限制root_dir、开启virtual_mode，生产环境优先SandboxBackend。
- 修订引擎兼容性：输出文档需在Word中验证修订痕迹/批注显示效果。
- 原文覆盖风险：修订前自动备份原始文件，避免不可逆修改。

十、里程碑与交付物
1) 方案确认：完成需求边界与专家视角模板。
2) 原型验证：对一份中等篇幅文档验证修订模式输出。
3) 长文档试运行：多章节断点续修与一致性回扫。
4) 交付：修订版docx + 修订摘要模板 + 运行说明。
5) 交付补充：AGENTS.md记忆模板与技能目录结构说明。
